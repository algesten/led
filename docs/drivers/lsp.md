# Driver: lsp

## Purpose

The LSP driver wraps one or more language-server subprocesses (e.g. `rust-analyzer`, `typescript-language-server`, `pyright-langserver`, `clangd`, `sourcekit-lsp`, `taplo`, `vscode-json-language-server`, `bash-language-server`) and exposes a narrow, typed interface to the model in both directions. It is simultaneously an **input driver** (for server-pushed notifications: `publishDiagnostics`, `$/progress`, `experimental/serverStatus`, `$/stderr`, `client/registerCapability`) and a **resource driver** (for request/response RPCs: goto-definition, completion, rename, code actions, formatting, inlay hints, and the pull-diagnostics cycle). All protocol-level concerns — JSON-RPC framing, request-id correlation, lsp-types conversion, URI/path mapping, server capability parsing, trigger-character detection, completion fuzzy filtering, push/pull diagnostic normalization, progress throttling, file-watcher registration, and prettier shelling-out — live inside `crates/lsp/` and never leak out. The model interacts with a single pair of streams: `Stream<LspOut>` (derived → driver, 15 variants) and `Stream<LspIn>` (driver → model, 9 variants). See `crates/lsp/src/lib.rs:14-200` for the full public surface; the manager loop lives at `crates/lsp/src/manager.rs:498-558`.

## Lifecycle

The driver is constructed once at startup by `led_lsp::driver(out_stream, server_override) -> Stream<LspIn>` (see `crates/lsp/src/lib.rs:204-231`). A single long-lived tokio task (`manager::run`, `crates/lsp/src/manager.rs:498`) runs the whole manager; it owns all server-process handles and protocol state. It bridges an `rx::Stream<LspOut>` into an mpsc channel (`out.on(...)` at `crates/lsp/src/lib.rs:210-214`) and an mpsc channel of results back into an `rx::Stream<LspIn>` (`tokio::task::spawn_local`, `crates/lsp/src/lib.rs:223-228`, with `yield_now()` between each push to avoid starving the reactive loop).

**Server-process lifecycle is per-language and lazy.** No server starts at driver construction time. `LspOut::Init { root }` arrives when the workspace resolves (derived `led/src/derived.rs:613-619`) and is recorded but does not spawn anything. The first `LspOut::BufferOpened { path, language, doc }` for a given `LanguageId` triggers `ensure_server_for_path` (`crates/lsp/src/manager.rs:757-795`), which looks up the `ServerConfig` in the registry (`crates/lsp/src/registry.rs:18-68` — a hard-coded table of eight languages) and spawns the server process in a new tokio task. Startup is fully async: two notification tasks are spawned per server (notification pump + stderr reader, `crates/lsp/src/server.rs:60-73`), the `initialize` request is sent (`crates/lsp/src/server.rs:222`), `initialized` + empty `workspace/didChangeConfiguration` notifications are pushed (`crates/lsp/src/server.rs:227-235`, the latter specifically because rust-analyzer waits for it before indexing). Any `BufferOpened` events that arrive before the server becomes ready are queued in `pending_opens` (`crates/lsp/src/manager.rs:815`) and flushed on `ManagerEvent::ServerStarted` (`crates/lsp/src/manager.rs:728-733`).

**Shutdown ordering matters.** When `Phase::Exiting` is reached the derived layer emits `LspOut::Shutdown` (`led/src/derived.rs:807-811`). The manager calls `shutdown_all` (`crates/lsp/src/manager.rs:802-806`), which on each server sends a JSON-RPC `shutdown` request, sleeps 100ms, sends the `exit` notification, then waits up to 5s for the child process before `kill()` (`crates/lsp/src/server.rs:296-323`). The child-process handle is held with `kill_on_drop(true)` as a safety net (`crates/lsp/src/server.rs:39`). The manager loop itself exits when either `cmd_rx` or `event_rx` yields `None` (`crates/lsp/src/manager.rs:537, 548`).

## Inputs (external → led)

The driver consumes four kinds of external input:

1. **JSON-RPC notifications from server stdout**, demultiplexed in `transport::spawn_reader` and re-routed as `ManagerEvent::Notification(language, LspNotification)` (`crates/lsp/src/manager.rs:776-780`, `crates/lsp/src/server.rs:86-92`). Handled methods: `textDocument/publishDiagnostics`, `$/progress`, `experimental/serverStatus`, `client/registerCapability`, `$/stderr` (synthetic, generated by the stderr reader when a line starts with "error"); all others are dropped with a debug log (`crates/lsp/src/manager.rs:1470-1472`).
2. **JSON-RPC responses** correlated via request id in `ResponseHandlers` (`crates/lsp/src/transport.rs:39-40`, `crates/lsp/src/server.rs:240-285`). Each outstanding request has a `oneshot::Sender` in a shared `HashMap<RequestId, ...>`.
3. **Filesystem events** from `notify::RecommendedWatcher`, installed lazily on `client/registerCapability` for `workspace/didChangeWatchedFiles` (`crates/lsp/src/manager.rs:1845-1927`). Events are matched against the server-supplied glob set, re-canonicalized, and re-emitted as `ManagerEvent::FileChanged(path, kind)` which becomes a `workspace/didChangeWatchedFiles` notification back to every running server (`crates/lsp/src/manager.rs:1929-1949`).
4. **Optional `prettier` subprocess output** when formatting a buffer in a Node project. `find_prettier` walks parents looking for `node_modules/.bin/prettier` (`crates/lsp/src/manager.rs:2102-2111`); if found, `run_prettier` pipes buffer contents through its stdin and parses the reformatted text as a whole-file replacement edit (`crates/lsp/src/manager.rs:2114-2168`). This bypasses the LSP for formatting while still returning through the same `LspIn::Edits` path.

## State owned by this driver

All of this is internal to `LspManager` (`crates/lsp/src/manager.rs:459-495`) and is not part of `AppState`:

- `registry: LspRegistry` — hard-coded language→command table with the optional `--test-lsp-server` override applied globally. (`registry.rs:3-92`)
- `servers: HashMap<LanguageId, Arc<LanguageServer>>` — one handle per running server.
- `root: CanonPath` — workspace root for `rootUri`.
- `pending_starts: HashSet<LanguageId>` — servers whose spawn is in flight.
- `opened_docs: HashSet<CanonPath>`, `pending_opens: HashSet<CanonPath>` — tracks which paths have had `didOpen` sent vs. which are waiting on server start.
- `docs: HashMap<CanonPath, Arc<dyn Doc>>` — the latest doc snapshot for every open path (used to compute content hashes, line-text for position conversion, and content for prettier).
- `languages: HashMap<CanonPath, Option<LanguageId>>` — pre-resolved per path (computed once in `BufferState::new`).
- `doc_versions: HashMap<CanonPath, i32>` — monotonic LSP document versions per path.
- `pending_code_actions: HashMap<CanonPath, Vec<CodeActionOrCommand>>` — raw code-action responses awaiting a `CodeActionSelect`.
- `completion_items: Vec<lsp_types::CompletionItem>`, `completion_path`, `completion_row`, `completion_prefix_start_col`, `completion_domain_items`, `completion_seq` — active completion session for re-filter, plus a monotonic sequence to discard stale responses.
- `progress_tokens: HashMap<String, ProgressState>` — active `$/progress` work-done tokens.
- `quiescent: HashMap<LanguageId, bool>` — latest `experimental/serverStatus` per language.
- `diag_source: DiagnosticSource` — the push/pull normalizer (mode, capabilities, `push_cache`, active `DiagWindow` with `hash_snapshot`, `pending_pulls`, `frozen`, `deadline`, `lsp_ready`, `init_delayed_request`). See `manager.rs:42-367`.
- `_file_watcher: Option<notify::RecommendedWatcher>` + `file_watcher_globs: Option<globset::GlobSet>` — installed lazily on `client/registerCapability`.
- `last_progress_sent: Instant`, `trigger_characters: Vec<String>`.

Inside each `LanguageServer`: `name`, `next_id: AtomicI32`, `outbound_tx` (stdin channel), `response_handlers: ResponseHandlers` (shared map of in-flight request ids to oneshot senders), `capabilities: Mutex<Option<ServerCapabilities>>`, `child: Mutex<Option<tokio::process::Child>>`. See `server.rs:15-23`.

## External side effects

- **Spawned subprocesses**: one per language, plus optionally `prettier` child processes during format. `kill_on_drop(true)` on the primary child; prettier children are awaited to completion.
- **stdout/stdin/stderr pipes** for each server; stderr is logged to the led log at warn level plus forwarded as `$/stderr` synthetic notifications for lines starting with "error" (`server.rs:64-72`).
- **Filesystem writes**: `apply_edits_to_disk` (`convert.rs`, called from `manager.rs:1520, 1540`) writes rename / code-action edits for files that are **not** currently opened in led. This is deliberate — open files round-trip through the model as `Edits` so the buffer's undo log captures them; closed files get direct disk writes.
- **Filesystem watcher**: installed via `notify::recommended_watcher` on the workspace root, recursive, lazy and bounded by the server's glob registration (`manager.rs:1884-1927`). On macOS this uses FSEvents (per project memory, FSEvents `watch()` is <3ms, so startup cost is negligible).

## Known async characteristics

- **Latency**: `initialize` varies widely (rust-analyzer indexing can take seconds; taplo/json-ls are near-instant). Per-request latency is bounded by server — completion responses are typically <10ms once warm; diagnostics pulls can be hundreds of ms on large projects. Prettier spawn adds a process fork plus its own runtime. The manager's only hard timeout is the pull-diagnostics 5s freeze deadline (`manager.rs:194, 353-361`).
- **Ordering**: Per-server JSON-RPC ordering is preserved by the writer task (single mpsc, one frame at a time, `transport.rs:42-68`). Responses can arrive out of order — id correlation picks the right handler. **Cross-server ordering is not guaranteed**: multiple servers run independently. Response-stream ordering into the model can also interleave with manager-local work (e.g. completion resolve and a later completion request). The completion `seq` counter (`manager.rs:493, 1003, 1587`) handles this specifically: older completion responses are dropped on arrival.
- **Cancellation**: Requests are never actively cancelled via LSP `$/cancelRequest`. Stale responses are tolerated and either (a) dropped by the seq check (completion) or (b) applied anyway (all other requests — the model-side chain in `lsp_of.rs` tolerates out-of-order replies because each producing stream emits one fine-grained `Mut`).
- **Backpressure**: `cmd_rx` is a bounded `mpsc::channel(64)` (`crates/lsp/src/lib.rs:206`); `try_send` in the rx-bridge silently drops on overflow (`crates/lsp/src/lib.rs:212`). Return channel is also bounded at 64. Internal `event_tx` is unbounded. During a pull-diagnostic freeze the manager specifically **does not** read `cmd_rx` (`manager.rs:533-543`), which creates intentional backpressure so edits don't interleave with the frozen cycle.

## State domain in new arch

Per `docs/rewrite/QUERY-ARCH.md:98-110`, `LspState` becomes its own domain atom owned by a single reducer. Current fields in `crates/state/src/lib.rs:1795-1813` map as follows:

```rust
pub struct LspState {
    pub diagnostics: HashMap<CanonPath, Loaded<(u64, Vec<Diagnostic>)>>,   // keyed by path, version-stamped
    pub inlay_hints: HashMap<CanonPath, Loaded<(u64, Vec<InlayHint>)>>,    // version-stamped
    pub completions: Loaded<CompletionSet>,                                // single session
    pub code_actions: Loaded<CodeActionPicker>,
    pub rename: Option<RenameState>,                                       // UI state; could live in UiState
    pub server_progress: HashMap<String, ServerProgress>,                  // keyed by server name
    pub trigger_chars_by_extension: HashMap<String, Vec<String>>,
    pub inlay_hints_enabled: bool,
}
```

Current `LspState.spinner_tick` is a timer-driven UI decoration and should move to `UiState`. Current `pending_request: Versioned<Option<LspRequest>>` disappears entirely — per the transition table in `DRIVER-INVENTORY-PLAN.md:227`, every `pending_*` `Versioned<T>` becomes a fire-and-forget `Request::*`. Current `pending_save_after_format` also disappears; the save pipeline in the new arch chains `Request::LspFormat` → on-completion `Request::SaveBuffer` explicitly.

Per-buffer diagnostics currently live on `BufferState.status.diagnostics` (`crates/state/src/lib.rs:123-227`) — a denormalized copy that stays rebased in lockstep with edits via `sync_annotations`. In the query arch, diagnostics live only in `LspState`, and the view layer derives a rebased per-buffer slice via a versioned query (see next section). The `Status::clear_diagnostics_on_row` / per-row shift / save-point replay logic in `buffer.rs` is replaced by a single pure `rebase_diagnostics(raw, edit_log_since_version)` function.

## Versioned / position-sensitive data

This is the hardest part of the driver to get right. Three `LspIn` variants carry position data that must be anchored to a specific buffer version:

- **Diagnostics** — already version-stamped today via `content_hash: PersistedContentHash` (`lib.rs:175-179`). The manager snapshots the hash at window-open time (`manager.rs:152-156, 220-227`), tags every forwarded `LspIn::Diagnostics` with it, and the model-side `offer_diagnostics` (`crates/state/src/lib.rs:1205-1254`) either accepts fast-path (hashes match), walks the undo log back to a save-point then forward through edits (replay path), or rejects. In the new arch this becomes `Loaded<(version, Vec<Diagnostic>)>` and `rebase_diagnostics(raw, edit_log_since_version)` (see `QUERY-ARCH.md:318-339`).
- **Inlay hints** — emitted without a version today (`lib.rs:180-183`). This is a gap. Hints are viewport-anchored, so a single edit inside the viewport invalidates them; the current UX gets away with it because the derived layer re-requests on every viewport/version change and replaces wholesale. In the new arch stamp them: `Event::LspInlayHints { path, version, hints }` and store as `Loaded<(u64, Vec<InlayHint>)>`.
- **Completions** — anchored to `(path, row, prefix_start_col)` and the monotonic `completion_seq`. The driver re-filters against the current buffer on every `BufferChanged` (`manager.rs:1735-1830`) and dismisses the session if the edit moves outside the identifier range. In the query arch model this stays as a `Loaded<CompletionSet>` keyed by session-id; the re-filter happens in a derived query rather than inside the driver.

Version-stamping happens **at source** in the driver: `DiagWindow.hash_snapshot` stores the hash at the moment the pull is issued (or the push is cached with the snapshot hash). Any subsequent `BufferChanged` that diverges from the snapshot closes the window and invalidates the cache (`manager.rs:588-592, 324-343`). This invariant is what makes rebase sound.

## Edge cases and gotchas

- **The freeze mechanism.** When a pull cycle opens in pull mode (`manager.rs:183-198`), the manager's main loop switches to "only read `event_rx`" (`manager.rs:533-543`), which stops draining `cmd_rx`. This intentionally backs up buffer edits so no `didChange` interleaves with in-flight `textDocument/diagnostic` calls. A 5s hard deadline (`manager.rs:194, 353-361`) cancels the freeze on server timeout — the model won't receive diagnostics but won't deadlock either. In the query arch, freezing should be expressed as a version barrier rather than a stopped channel: the scheduler refuses to pick up new buffer-change work while a diagnostic pull is in flight, or the pull dispatcher rebases the result against edits that have happened since.
- **Pull-only diagnostic policy.** Per project memory (`feedback_pull_diagnostics`): "Always pull diagnostics, never rely on publishDiagnostics push. Freeze mechanism controls when diagnostics are accepted." The current code still handles push (`manager.rs:233-272`) and even switches mode from pull to push if a server pushes anyway (`manager.rs:234-245`) — this is defensive because real servers (rust-analyzer, clangd) do push by default. The new arch should either (a) disable `publishDiagnostics` capability in `initialize` entirely and force pull, or (b) keep push as an internal validation cache but never project it into `LspState`. The cache-merge logic (`manager.rs:305-319` — pull is source of truth for which diagnostics, push is source of truth for detail) encodes the current trade-off.
- **Multi-root / multi-server handling.** Each `LanguageId` gets at most one server. If the workspace contains multiple `.rs` files under different Cargo manifests, they all talk to one rust-analyzer rooted at the workspace root (`server.rs:35`). This is a design choice — not all real-world multi-root setups work. The registry is flat and doesn't support per-directory overrides. [unclear — will the new arch preserve this single-server-per-language constraint or generalize to N servers keyed by root?]
- **Server spawn failure.** `NotFound` (binary not on PATH) is logged at info and emits `ServerError { not_found: true, .. }` which is silently swallowed (`manager.rs:736-742`) — no alert. Other spawn errors emit `LspIn::Error`. RPC errors during startup (e.g. `initialize` fails) propagate as `LspError` from `request` and are surfaced as `LspIn::Error` through the normal path.
- **Server crash mid-session.** If the server process exits, subsequent writes to its `outbound_tx` fail silently in the writer task (`transport.rs:57-65`); response futures never resolve and are leaked (no timeout). Current code has no auto-restart. The model sees nothing until the next request also silently fails. [unclear — should the driver detect child exit and emit an `LspIn::Error` + scrub state? Currently it does not.] The new arch should treat server death as an explicit `Event::LspServerDied { server }` with an idle→unloaded transition for its `Loaded<_>` slots.
- **`client/registerCapability` handling.** Only `workspace/didChangeWatchedFiles` is actually implemented (`manager.rs:1458-1460, 1845-1882`); all other dynamic capabilities are ignored. rust-analyzer and TypeScript both use this to register their file watchers post-init, so the mechanism is load-bearing. The glob set is only installed **once** (first registration wins — subsequent calls replace it). No matching `client/unregisterCapability` handling exists.
- **`completion_seq` protects against stale responses but not stale state.** If the user types `fo` (req seq=1), then `ob` (req seq=2), response 2 arrives first and sets the session; response 1 arrives second and is dropped (`manager.rs:1587-1589`). But if response 2 is lost and response 1 arrives alone, it will be applied and immediately re-filtered against the now-`fob` doc; if `fob` doesn't match any item, the session clears silently.
- **Prettier edits bypass the LSP's version tracking.** The prettier path emits a whole-file replacement `TextEdit` (`manager.rs:2161-2167`) with no version stamp; it's applied straight through `LspIn::Edits`. This is safe because format-on-save runs with no concurrent edits, but the new arch should stamp format edits with the version they were computed against so the reducer can reject stale format results.
- **Empty-edits `Edits` event is a control signal.** `RequestResult::FormatDone` emits `LspIn::Edits { edits: vec![] }` (`manager.rs:1625-1627`) specifically so the model-side `format_done_*` streams in `lsp_of.rs:86-107` fire and trigger the save. This overloading of `Edits` as both "here are edits" and "format is done" is load-bearing — the new arch should split them: `Event::LspEdits { edits }` vs. `Event::LspFormatComplete { path }`.
- **Line-text lookup at position-conversion time.** Converting between LSP UTF-16 positions and led's UTF-8 character offsets requires the current line text, obtained either from the cached doc or (for pre-open pushes) by reading the file from disk (`manager.rs:1345-1354`). If the file on disk has drifted from the buffer, UTF-16 offsets will be wrong. This matters only for the narrow window between a buffer being opened and its first `didChange`.
- **`BufferClosed` is never emitted.** See the note in the outputs table — derived layer suppresses it (`led/src/derived.rs:650-651`). The driver relies on its own fs-watcher for file-gone detection. Dead code.

### Fake-lsp extensions required to unlock the "missing" column

Many scenarios above depend on extending `fake-lsp` (see also `driver-events.md:454`). The high-value additions are:

1. **Delayed progress / hold-busy mode** — to test `spinner`, quiescence gating, busy→idle transitions.
2. **Inlay-hints response** — currently always `[]`.
3. **`experimental/serverStatus` emission** — to test the quiescence-deferred RequestDiagnostics path.
4. **Out-of-order response injection** — to test completion seq filtering.
5. **Dynamic `client/registerCapability` emission with watchers** — to test the fs-watch bridge.
6. **Mid-session stderr write** — to test `$/stderr` → `LspIn::Error`.
7. **Exit-on-trigger** — to test server death handling.
8. **Push-mode-only response variants** — to test the mode-switch path from pull to push.
9. **Per-language behavior** (currently fake-lsp is one binary for all `LanguageId`s via `--test-lsp-server`) — to test multi-server scenarios.

Plus two runner-level additions:
- **Virtual clock** (already planned per `scenario.rs:68` and `driver-events.md:433`) — for 5s pull timeout, 200ms progress throttle, spinner.
- **Mid-test filesystem mutation** (per `driver-events.md:431`) — for the register-capability watcher bridge.
